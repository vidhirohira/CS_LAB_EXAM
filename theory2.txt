PAPER 1
Theory: Explain why ECB mode leaks structural information and under what type of
data this weakness becomes visible.

Answer (Expanded):

ECB (Electronic Codebook) mode encrypts each identical plaintext block into an
identical ciphertext block using the same key. This is because ECB processes each
block independently without any chaining mechanism. Mathematically, if plaintext
block Pi = Pj, then ciphertext block Ci = Cj = Encrypt(K, Pi).

Why structural information leaks:
1. Repeating patterns in plaintext remain visible as repeating patterns in ciphertext.
2. An attacker can perform block-wise analysis without decryption.
3. The encryption does not hide data patterns — it acts like a codebook lookup.
4. For example, in an encrypted bitmap image, solid color backgrounds produce
   repeating ciphertext blocks, revealing the outline of figures and shapes.

Types of data where weakness becomes clearly visible:
- Images with large uniform areas (logos, icons, diagrams, scanned documents)
- Database columns with repeated values (country codes, status fields, ZIP codes)
- Formatted documents with repeated headers/footers or template structures
- Source code with repeated syntax patterns or comment blocks
- Encrypted passwords stored per block (vulnerable to codebook attacks)
- Encrypted financial data with repeated transaction amounts

Real-world impact: The famous "ECB Penguin" image demonstrates how the Linux
kernel's cryptographic API's ECB encryption of a Tux logo still reveals the
penguin's shape despite encryption. An attacker can identify relationships
between blocks without decrypting, making ECB unsuitable for any data with
discernible structure. NIST SP 800-38A explicitly recommends against ECB for
any application encrypting more than one block of data. ECB should only be used
for encrypting single-block messages like a single cryptographic key.

PAPER 2
Theory: Justify how IV randomness contributes to semantic security in block
ciphers.

Answer (Expanded):

Semantic security means that an attacker cannot distinguish ciphertexts of two
plaintexts, even if they choose the plaintexts. More formally, no probabilistic
polynomial-time adversary can gain any information about the plaintext from the
ciphertext beyond its length.

How IV randomness helps:
A random, unpredictable IV (Initialization Vector) ensures that encrypting the
same plaintext twice produces completely different ciphertexts, breaking any
deterministic relationship. For CBC mode specifically:
- The IV is XORed with the first plaintext block before encryption
- A random IV ensures the first ciphertext block is unique
- This uniqueness propagates through all subsequent blocks due to chaining
- Even if the same plaintext is encrypted a million times, ciphertexts look random
- Attackers cannot tell whether two ciphertexts correspond to same or different
  plaintexts

Without a random IV (e.g., fixed IV or counter):
- Identical plaintexts yield identical ciphertexts
- Attackers can detect repeating messages
- This violates semantic security and enables frequency analysis
- It also enables chosen-plaintext attacks where attacker sees deterministic
  encryption of guessed plaintexts

Additional requirement: IV must be unpredictable (not just unique). For CBC mode,
a predictable IV enables adaptive chosen-plaintext attacks. For CTR mode, IV must
never repeat with the same key (counter uniqueness suffices, randomness not
strictly required). NIST recommends using a cryptographically secure random
number generator for IV generation.

PAPER 3
Theory: Explain the error propagation behavior in CFB mode and its impact on
reliability.

Answer (Expanded):

In CFB (Cipher Feedback) mode, the ciphertext is fed back into the shift register
to generate the keystream. A single-bit error in the ciphertext affects two
categories of decrypted plaintext:

First category - Direct effect (corresponding plaintext bit):
- The bit in the same position within the current plaintext block is flipped
- This happens because ciphertext is XORed with keystream to produce plaintext
- Error affects exactly that bit position and no other bits in current block

Second category - Propagating effect (subsequent blocks):
- The corrupted ciphertext block shifts into the shift register
- It affects all subsequent plaintext blocks until it shifts completely out
- For CFB-128: error affects current block + next 128 bits (one full block)
- For CFB-8: error affects current byte + next 8 bytes (one full block)
- For CFB-1 (rare): error affects current bit + next 8 bits

Detailed propagation mechanics:
- After shift register receives corrupted ciphertext, next N bits of keystream are corrupted
- This continues for exactly (block_size / segment_size) subsequent blocks
- In CFB-128 with 128-bit block: total error window = current block + 1 subsequent block
- In CFB-8 with 128-bit block: total error window = current byte + 15 subsequent bytes
- After corrupted data shifts out (entire shift register content is replaced),
  decryption resynchronizes automatically

Impact on reliability:
- CFB is self-synchronizing — after a limited number of blocks (typically 1-2 blocks),
  decryption recovers correctly without external intervention
- This makes CFB suitable for streaming applications over unreliable channels
- Burst errors (multiple consecutive bit errors) are handled gracefully
- However, CFB provides no error detection or correction on its own
- Not suitable for integrity-sensitive data without additional authentication (like MAC)
- Used in older SSL/TLS versions and some real-time encryption systems

Comparison with other modes:
- ECB/CBC: error corrupts whole block but no recovery (block aligned)
- CFB: error corrupts current + next block, then recovers
- OFB: error corrupts only current bit, never recovers (no self-sync)
- CTR: error corrupts only current bit, never recovers

PAPER 4
Theory: Explain why OFB mode limits error propagation and where it is preferred.

Answer (Expanded):

OFB (Output Feedback) mode generates a keystream independently of both plaintext
and ciphertext. The keystream is produced by repeatedly encrypting the IV and
then feeding the output back into the encryption function (feedback does NOT
include ciphertext). Mathematically:
- Keystream block S0 = Encrypt(K, IV)
- S1 = Encrypt(K, S0), S2 = Encrypt(K, S1), etc.
- Ciphertext Ci = Plaintext Pi XOR Si
- Decryption: Pi = Ciphertext Ci XOR Si

Why error propagation is limited:
- A single-bit error in ciphertext affects only the corresponding bit of the
  decrypted plaintext at that exact position
- The error does NOT affect any other bits in the same block
- The error does NOT affect any subsequent blocks
- This is because the keystream generation uses only the IV and key — ciphertext
  is never fed back into the encryption function
- The keystream remains identical for both encryption and decryption regardless
  of ciphertext corruption

Important limitation - No self-synchronization:
- Unlike CFB, OFB does NOT recover from lost or inserted bits
- If synchronization is lost (bit insertion/deletion), decryption fails permanently
- This makes OFB unsuitable for channels that may insert or drop bits
- Requires byte/block alignment to be maintained by lower layers

Where OFB is preferred:
1. Satellite communications: high bit-error rates but no bit insertion issues
2. Fiber optic links: low noise but burst errors tolerated better with OFB
3. Noisy channels where burst errors are common but you want to avoid error spreading
4. Applications where error localization is preferred over error propagation
5. Pre-encryption for storage systems (single-bit errors don't cascade)
6. Real-time voice/video where localized glitch is acceptable but avalanche is not

Comparison table:
+----------------+------------------+------------------+-------------------+
| Mode           | Error affects    | Self-synchroniz. | Preferred usage   |
+----------------+------------------+------------------+-------------------+
| OFB            | 1 bit only       | No               | Satellite, fiber  |
| CFB            | Current+next     | Yes (1-2 blocks) | Streaming, noisy  |
| CBC            | Entire block     | No               | Bulk file encrypt |
| ECB            | Entire block     | No               | Single block only |
| CTR            | 1 bit only       | No               | Parallel, random  |
+----------------+------------------+------------------+-------------------+

Security note: OFB keystream repeats after 2^(n/2) blocks for n-bit block cipher,
so a fresh IV/key pair must be used frequently enough to avoid keystream reuse.

PAPER 5
Theory: Explain why CBC hides patterns better than ECB using block dependency
reasoning.

Answer (Expanded):

CBC (Cipher Block Chaining) hides patterns through a simple but powerful
mechanism: each plaintext block is XORed with the previous ciphertext block
BEFORE encryption. Mathematically:
- C0 = Encrypt(K, P0 XOR IV)
- C1 = Encrypt(K, P1 XOR C0)
- C2 = Encrypt(K, P2 XOR C1)
- General: Ci = Encrypt(K, Pi XOR Ci-1)

Why this creates block dependency:
1. Changing one plaintext block (Pi) changes its ciphertext block (Ci) completely
2. Changed Ci then affects encryption of Pi+1 (since Pi+1 XOR Ci changes)
3. Changed Ci+1 affects Pi+2, and so on — cascade continues to end of message
4. Therefore, a single-bit change in Pi changes ALL subsequent ciphertext blocks
5. Dependency is one-way forward; previous blocks unaffected by later changes

How CBC hides identical plaintext blocks:
- Suppose Pi = Pj for i ≠ j
- For ECB: Ci = Cj directly, revealing the equality
- For CBC: Ci = Encrypt(K, Pi XOR Ci-1)
- Cj = Encrypt(K, Pj XOR Cj-1)
- Since Ci-1 ≠ Cj-1 (practically guaranteed with random IV and chaining),
  the inputs to encryption are different unless by improbable coincidence
- Therefore Ci ≠ Cj even though Pi = Pj
- No repeating patterns visible in ciphertext

Real-world demonstration:
Encrypt "AAAAAAAABBBBBBBB" in ECB and CBC with same key:
- ECB output: [C1][C1][C2][C2] (repeating A block gives same C1, repeating B block gives same C2)
- CBC output: [C1][C2][C3][C4] (all different, no visible pattern)

Why hiding patterns matters:
- Prevents attackers from seeing message structure (headers, footers, repeating fields)
- Prevents identification of duplicate data blocks
- Prevents codebook attacks where attacker builds dictionary of plaintext->ciphertext
- Provides semantic security (indistinguishability under chosen-plaintext attack)

Limitation of CBC: Requires padding because plaintext must be multiple of block size.
This can lead to padding oracle attacks if error messages leak padding validity.

Comparison summary:
+-------------+--------------------------+--------------------------+
| Aspect      | ECB                      | CBC                      |
+-------------+--------------------------+--------------------------+
| Dependency  | None (parallel)          | Sequential (chained)     |
| Pattern     | Visible (same blocks)    | Hidden (all different)   |
| Parallelism | Fully parallel           | Encryption sequential    |
| IV required | No                       | Yes (random, one block)  |
| Security    | Weak (pattern leakage)   | Strong (semantic secure) |
+-------------+--------------------------+--------------------------+

PAPER 6
Theory: Explain the role of key pairs in RSA and why incorrect key usage fails
mathematically.

Answer (Expanded):

RSA key pairs consist of a public key (e, n) for encryption/verification and a
private key (d, n) for decryption/signing, where:
- n = p × q (product of two large distinct primes)
- φ(n) = (p-1) × (q-1) (Euler's totient function)
- e is chosen (typically 65537) such that 1 < e < φ(n) and gcd(e, φ(n)) = 1
- d is computed as the modular multiplicative inverse: e × d ≡ 1 (mod φ(n))
- This means e × d = 1 + k × φ(n) for some integer k

Encryption and decryption mathematically:
- Encryption: c = m^e mod n (using public key)
- Decryption: m = c^d mod n (using private key)

Why correct key pair works (proof):
c^d mod n = (m^e)^d mod n = m^(e×d) mod n = m^(1 + k×φ(n)) mod n
By Euler's theorem: m^(φ(n)) ≡ 1 mod n when gcd(m,n)=1
Therefore: m^(1 + k×φ(n)) = m × (m^(φ(n)))^k ≡ m × 1^k = m mod n
For m not coprime to n (rare case of being multiple of p or q), the equation
still holds by Chinese Remainder Theorem.

Why using the wrong key fails:
Case 1: Encrypt with private key instead of public key:
- Encrypting with (d,n) gives c = m^d mod n
- Attempting to decrypt with (e,n) gives m' = c^e mod n = m^(d×e) mod n
- Since d×e = 1 + k×φ(n), this actually works mathematically because multiplication
  is commutative! This is why RSA encryption and signing use same operation.
- However, using private key for general encryption is UNSAFE because:
  * Private key may be weaker or expose timing channels
  * Anyone with public key (everyone) can decrypt
  * No authentication purpose served

Case 2: Decrypt with wrong private key (d2,m2) that doesn't match:
- c = m^e mod n1 (encrypted with (e1,n1))
- m' = c^d2 mod n2 gives garbage because n1 ≠ n2 and modular arithmetic mismatches
- Even if n1=n2 but d2 not the true inverse of e1, Euler's theorem fails
- Result is random-looking number, not the original message

Case 3: Decrypt with correct d but modified n:
- If n is changed but d calculated incorrectly, no mathematical inverse exists
- Decryption produces meaningless output

Role of key pairs summarized:
+----------------+-------------------+------------------+-------------------+
| Operation      | Key used          | Security property| Mathematical basis|
+----------------+-------------------+------------------+-------------------+
| Encryption     | Public (e,n)      | Confidentiality  | Trapdoor function |
| Decryption     | Private (d,n)     | Confidentiality  | Euler's theorem   |
| Signing        | Private (d,n)     | Authenticity     | Same as decryption|
| Verification   | Public (e,n)      | Integrity        | Same as encryption|
+----------------+-------------------+------------------+-------------------+

PAPER 7
Theory: Explain how hybrid encryption balances performance and security in
real-world systems.

Answer (Expanded):

Hybrid encryption solves the core dilemma of cryptography: symmetric ciphers are
fast but have key distribution problems, while asymmetric ciphers solve key
distribution but are slow for bulk data.

The Problem:
- AES-256 encrypts at >1 GB/s on modern hardware
- RSA-2048 encrypts at <100 KB/s (10,000x slower)
- Encrypting a 1 GB file with RSA would take hours
- Also, RSA ciphertext expands (output size = key size, not message size)
- But distributing a shared AES key securely over the internet is impossible
  without already having a secure channel

Hybrid encryption solution (step-by-step):

Step 1: Generate a random session key (symmetric key):
- Typically 128, 192, or 256 bits for AES
- Generated using cryptographically secure random number generator
- Used only for this single encryption session (ephemeral)

Step 2: Encrypt the actual data using symmetric cipher:
- Use AES-CBC, AES-GCM, or AES-CTR with the session key
- Fast: gigabytes per second
- Output size = plaintext size + padding + IV overhead
- No practical limit on data size

Step 3: Encrypt the session key using asymmetric cipher:
- Use recipient's RSA public key to encrypt only the session key (128-256 bits)
- Output size = key size (2048 bits for RSA-2048) — fixed, small overhead
- Slow but acceptable because input is tiny (a few hundred bits)

Step 4: Send both together:
- Final package = [Encrypted_session_key] + [IV] + [Ciphertext]
- Recipient decrypts session key with their private key (fast)
- Then decrypts bulk data with AES (fast)

Real-world usage examples:
1. TLS/HTTPS: Client generates random key, encrypts with server's RSA public key,
   sends to server, then both use that key for AES session encryption
2. PGP/GPG: Generates random session key for each file, encrypts file with AES,
   encrypts session key with recipient's RSA key
3. S/MIME: Same as PGP for email encryption
4. SSH: Hybrid key exchange (Diffie-Hellman + AES)
5. Signal Protocol: Double Ratchet uses hybrid approach per message

Performance comparison:
+------------------------+---------------+------------------+
| Operation              | Time (approx) | Acceptable for    |
+------------------------+---------------+------------------+
| AES encrypt 1GB        | <1 second     | Any file size     |
| RSA encrypt 1GB        | >2 hours      | No                |
| RSA encrypt 256-bit key|<1 millisecond | Yes (tiny input)  |
| Hybrid (AES+RSA) for 1GB|<1 second + <1ms| Excellent        |
+------------------------+---------------+------------------+

Security advantages:
- Session keys are ephemeral (forward secrecy if deleted after use)
- RSA private key never used for bulk encryption (reduces exposure)
- Compromise of RSA key doesn't reveal past sessions if session keys deleted
- Can combine multiple asymmetric algorithms (e.g., RSA + ECC) for redundancy

PAPER 8
Theory: Explain avalanche effect quantitatively (in terms of bit changes).

Answer (Expanded):

The avalanche effect is a fundamental property of cryptographic algorithms where
a small change in input (plaintext or key) causes a large, seemingly random
change in the output (ciphertext or hash). Ideally, flipping a single input bit
should cause each output bit to change with 50% probability.

Quantitative measurement:

Strict Avalanche Criterion (SAC):
- For each output bit j, when any single input bit i is flipped, the probability
  that output bit j changes should be exactly 1/2
- Mathematically: P(Output_j changes | Input_i flipped) = 0.5 for all i,j

Bit changes per block:
- Block size = B bits
- Let H = Hamming distance between two ciphertexts (number of differing bits)
- For ideal cipher: E[H] = B/2
- Standard deviation ideal: sqrt(B)/2
- For AES-128: B = 128, ideal H = 64 bits differ
- For AES-256: B = 128, ideal H = 64 bits differ (same block size)

Avalanche effect measurement process:
1. Generate random plaintext P and key K
2. Compute ciphertext C = Encrypt(K, P)
3. Flip exactly one bit in plaintext at position i → get P'
4. Compute ciphertext C' = Encrypt(K, P')
5. Count how many bits differ between C and C' → H_i
6. Repeat for all i from 1 to B (or many random positions)
7. Average H_i across all tests
8. Ideal result: H_avg = B/2 = 64 for AES

Acceptable range:
- Good cipher: 50% ± 5% (i.e., 60-68 bits for AES)
- Weak cipher: <10% or >90% indicates bias
- AES measured values: typically 63-65 bits per single-bit change

Avalanche in hash functions (SHA-256):
- Input block size varies, output fixed at 256 bits
- Flipping one input bit should cause ~128 output bits to change
- Measured SHA-256: 126-130 bits typical

Why avalanche effect matters:
1. Prevents attackers from predicting outputs based on small input changes
2. Ensures ciphertexts of similar plaintexts look completely unrelated
3. Makes differential cryptanalysis more difficult
4. Is a necessary (but not sufficient) condition for security

Example calculation:
If B=128, ideal H=64:
- Avalanche percentage = (H/B) × 100% = (64/128) × 100% = 50%
- If measured H=32 → 25% avalanche (weak diffusion)
- If measured H=96 → 75% avalanche (biased output)

Comparison across algorithms:
+------------------+--------+----------+-------------------+
| Algorithm        | Block  | Ideal H  | Achieved H (meas.)|
+------------------+--------+----------+-------------------+
| AES-128          | 128    | 64       | 63-65 (excellent) |
| DES              | 64     | 32       | 30-34 (good)      |
| 3DES             | 64     | 32       | 30-34 (good)      |
| Blowfish         | 64     | 32       | 31-33 (excellent) |
| RC4 (stream)     | 8 bits | 4 bits   | 3-5 (variable)    |
+------------------+--------+----------+-------------------+

PAPER 9
Theory: Explain why modern systems prefer SHA-256 over MD5.

Answer (Expanded):

MD5 (Message Digest 5) was designed in 1991 by Ron Rivest. SHA-256 (Secure Hash
Algorithm 256-bit) was published by NIST in 2001. Modern systems prefer SHA-256
for multiple critical security reasons:

1. Collision Resistance (Most Critical):
   - MD5: Collisions can be generated in seconds on a standard laptop
   - First practical collision (2004): 2^69 operations
   - 2008: Researchers generated two different SSL certificates with same MD5 hash
   - 2012: Flame malware used MD5 collision to forge Windows code-signing certificate
   - 2013: 18 minutes to generate MD5 collision on a single PC
   - SHA-256: No practical collisions known despite 23+ years of analysis
   - Best known attack (as of 2024): 2^65.5 operations (impractical)

2. Pre-image Resistance:
   - MD5: Pre-image attack reduced from 2^128 to 2^123.4 (theoretical)
   - Second pre-image attacks have been shown in weakened variants
   - SHA-256: Full 2^256 pre-image resistance intact
   - No known attack better than brute force for full-round SHA-256

3. Output Length:
   - MD5: 128 bits (16 bytes) → birthday attack strength = 2^64
   - 2^64 operations is within reach of nation-states/corporations
   - SHA-256: 256 bits (32 bytes) → birthday attack strength = 2^128
   - 2^128 operations remains computationally infeasible for foreseeable future

4. Structural Security:
   - MD5 uses a 512-bit block size and 128-bit state
   - Weakened by known differential cryptanalysis
   - The compression function has been broken
   - SHA-256 uses 512-bit block size with 256-bit state
   - Designed with stronger diffusion and more rounds (64 vs MD5's 64 but better design)

5. Real-world Attacks Exploiting MD5 Weaknesses:
   - Rogue CA certificates (2008): Attackers created fake certificate authority cert
   - Flame malware (2012): Forged Microsoft Windows update signature
   - Fake SSL certificates used in MITM attacks
   - Password cracking: MD5 is too fast (millions/sec on GPUs)
   - File integrity: Two different files can have same MD5

6. NIST Deprecation Timeline:
   - 2004: MD5 declared "cryptographically broken"
   - 2008: NIST formally deprecates MD5 for digital signatures
   - 2012: RFC 6151 highlights MD5 security risks
   - 2015: Microsoft and other vendors stop accepting MD5 certificates
   - Current: MD5 only permitted for non-cryptographic uses (checksumming)

Comparison table:
+----------------------+------------------------+------------------------+
| Property             | MD5                    | SHA-256                |
+----------------------+------------------------+------------------------+
| Output size          | 128 bits               | 256 bits               |
| Collision found?     | Yes (trivial, seconds) | No (practical unknown) |
| Birthday attack cost | 2^64 operations        | 2^128 operations       |
| Speed (GB/s)         | ~2 GB/s (very fast)    | ~0.5 GB/s (still fast) |
| NIST approved?       | No (deprecated)        | Yes                    |
| Use case today       | Non-crypto checksums   | Digital signatures, TLS|
+----------------------+------------------------+------------------------+

When MD5 is still acceptable (non-cryptographic):
- Basic file integrity check (if tampering not a concern)
- Non-security checksums for data deduplication
- Historical/legacy system compatibility only

PAPER 10
Theory: Explain the concept of pre-image resistance and its importance.

Answer (Expanded):

Pre-image resistance (also called the one-way property) is a fundamental security
property of cryptographic hash functions. It can be broken into three distinct
but related concepts:

First Pre-image Resistance:
- Given a hash value h, it is computationally infeasible to find any message m
  such that hash(m) = h
- Formal: For any probabilistic polynomial-time adversary A, probability that
  A(h) returns m with hash(m)=h is negligible
- Security level: O(2^n) operations for n-bit hash (e.g., 2^256 for SHA-256)
- Also called "one-wayness"

Second Pre-image Resistance:
- Given a specific message m1, it is infeasible to find a different message
  m2 (m2 ≠ m1) such that hash(m2) = hash(m1)
- Security level: O(2^n) operations (harder than collision resistance)
- This is stronger than first pre-image resistance

Difference from collision resistance:
- Collision resistance: Find any two messages m1≠m2 with same hash
- Pre-image resistance (first): Given hash, find one input
- Pre-image resistance (second): Given input, find another with same hash

Why Pre-image Resistance is Critically Important:

1. Password Storage (Most Common Use):
   - System stores hash(password) instead of password
   - Attacker who breaches database only gets hashes
   - Pre-image resistance prevents attacker from recovering passwords from hashes
   - However, weak passwords still vulnerable to rainbow tables/dictionary attacks
   - Hence password hashing uses iteration (bcrypt, PBKDF2, Argon2)

2. Digital Signatures:
   - Signer hashes message, then signs the hash (not the message)
   - Without pre-image resistance, attacker could:
     * Find any message m2 that has same hash as valid message m1
     * Submit m2 for signature (but actually get signature valid for m1)
   - Pre-image resistance (second) prevents this substitution

3. Commitment Schemes:
   - Alice commits to value v by publishing hash(v)
   - Later reveals v to Bob, who verifies hash(v) matches
   - Pre-image resistance ensures Alice cannot find a different v' with same hash
   - Prevents cheating in auctions, voting, secure coin flips

4. Message Integrity without Key:
   - Publish hash of file (e.g., software download)
   - User downloads file, recomputes hash, compares
   - Pre-image resistance prevents attacker from:
     * Finding malicious file with same hash as legitimate file
   - Prevents substitution attacks

5. Proof-of-Work (Blockchain, Bitcoin):
   - Miners search for nonce such that hash(block) < target
   - Pre-image resistance ensures finding nonce requires brute force
   - Can't reverse-engineer nonce from target hash

Mathematical Definitions:

Let H: {0,1}* → {0,1}^n be a hash function.
Let Adv be an attacker with runtime t.

Pre-image resistance:
P[ H(Adv(h)) = h ] < ε(t) for negligible ε, where h = H(m) for random m

Second pre-image resistance:
P[ Adv(m) = m' and m' ≠ m and H(m') = H(m) ] < ε(t)

Example of Broken Pre-image (Weakened MD5):
- MD5 previously had 2^128 security
- Attacks reduced first pre-image to 2^123.4 (small reduction, not catastrophic)
- But MD5 is still considered broken for pre-image in practice due to collisions

Security Levels:
+------------+--------------+-------------------+-------------------+
| Hash       | Bits (n)     | First Pre-image   | Second Pre-image  |
+------------+--------------+-------------------+-------------------+
| MD5        | 128          | 2^123.4 (theor.)  | 2^123.4           |
| SHA-1      | 160          | 2^159 (theor.)    | 2^159             |
| SHA-256    | 256          | 2^256 (intact)    | 2^256 (intact)    |
| SHA-512    | 512          | 2^512 (intact)    | 2^512 (intact)    |
+------------+--------------+-------------------+-------------------+

PAPER 11
Theory: Explain why strong collision resistance is computationally difficult to
break.

Answer (Expanded):

Collision resistance is the property that it should be computationally infeasible
to find any two distinct inputs x ≠ y such that hash(x) = hash(y). Strong
collision resistance remains difficult to break for well-designed hash functions
like SHA-256 due to several mathematical and computational factors.

The Birthday Paradox (Why 2^(n/2) matters):

Fundamental bound:
- For an n-bit hash function, there are 2^n possible outputs
- By pigeonhole principle, collisions MUST exist (2^n inputs → only 2^n outputs)
- But finding them is the challenge
- Birthday paradox: With approximately sqrt(2^n) = 2^(n/2) random inputs,
  probability of at least one collision is > 1/2
- For n=256: 2^(128) inputs needed (not 2^256)
- This is the best possible generic attack on any hash function

Why 2^128 is still infeasible:

1. Storage Requirements:
   - To run birthday attack, you must store 2^64 (NOT 2^128) pairs due to Floyd's cycle-finding
   - But still: 2^64 entries × (hash + input) ≈ 2^64 × (32 + 32) bytes ≈ 2^70 bytes ≈ 1 billion GB
   - This is impossible with any existing or foreseeable storage technology

2. Time Requirements:
   - 2^128 operations at 1 trillion operations per second = 10^22 seconds
   - Age of universe ≈ 4.3 × 10^17 seconds
   - Would take ~23,000 times the age of the universe
   - Even with massive parallelization (1 million devices each doing 1 trillion ops/sec):
     * Still 10^16 seconds ≈ 300 million years

3. Computational Comparisons (Energy):
   - One SHA-256 operation ≈ 0.5 microjoules
   - 2^128 operations ≈ 1.7 × 10^22 joules
   - Annual global energy production ≈ 5 × 10^20 joules
   - Would require ~34 years of ALL EARTH'S ENERGY

4. No Mathematical Shortcut Known:
   - Unlike MD5 (where differential cryptanalysis reduces complexity to 2^18.4 operations),
     SHA-256 has no known practical attack better than 2^128
   - Merkle-Damgård construction with Davies-Meyer compression function
   - 64 rounds provide extensive diffusion
   - No feasible linear or differential paths found

Why broken hashes (MD5, SHA-1) weakened:

MD5 collision attack (Wang, 2004):
- Complexity reduced from 2^64 to 2^39 operations (later to 2^18.4 for chosen prefix)
- Exploited specific differential characteristics in MD5's compression function
- Weak message expansion and insufficient non-linearity

SHA-1 (SHAttered attack, 2017):
- Complexity 2^63.1 vs ideal 2^80 (still large but feasible with Google Compute Engine)
- Costs $110,000 USD of cloud computing to generate collision
- Chosen-prefix collision now possible

Side-by-side comparison:
+----------------+--------------+----------------+----------------------+
| Hash           | Bits (n)     | Ideal break    | Actual best attack   |
+----------------+--------------+----------------+----------------------+
| MD5            | 128          | 2^64           | 2^18.4 (trivial)     |
| SHA-1          | 160          | 2^80           | 2^63.1 (expensive)   |
| SHA-256        | 256          | 2^128          | 2^128 (no reduction) |
| SHA-512        | 512          | 2^256          | 2^256 (no reduction) |
+----------------+--------------+----------------+----------------------+

Why quantum computers don't fully break collision resistance:
- Grover's algorithm speeds up brute force (2^(n/2) → 2^(n/3) for pre-image)
- But for collision: Brassard-Høyer-Tapp algorithm gives 2^(n/3) vs classical 2^(n/2)
- For SHA-256: 2^85.3 quantum operations still infeasible
- No quantum algorithm fundamentally breaks collision resistance structure

Practical implications:
- Strong collision resistance ensures digital signatures remain unforgeable
- Prevents substitution attacks on certificates and contracts
- Enables secure file fingerprinting (e.g., Git uses SHA-256)
- Backbone of blockchain and cryptocurrency mining
- No practical collision for SHA-256 expected for decades

PAPER 12
Theory: Explain how HMAC prevents tampering compared to plain hashing.

Answer (Expanded):

HMAC (Hash-based Message Authentication Code) is a mechanism for message
authentication using cryptographic hash functions. It provides both integrity
and authenticity, unlike plain hashing which provides only a non-authenticated
checksum.

Plain Hashing Problem:
- hash(message) is deterministic and publicly computable
- Anyone can calculate the hash of any message
- Attacker modifies M to M' and recomputes hash(M') easily
- Receiver cannot distinguish original from modified message
- Plain hash provides zero tamper protection against active attackers

HMAC Construction (RFC 2104):
HMAC(K, m) = H( (K ⊕ opad) || H( (K ⊕ ipad) || m ) )

where:
- H is the hash function (SHA-256, SHA-512, etc.)
- K is the secret key padded to block size (B bytes)
- ipad = 0x36 repeated B times
- opad = 0x5C repeated B times
- || denotes concatenation

Step-by-step processing:
1. If key K is longer than block size B, hash it first: K = H(K)
2. Pad K with zeros to length B
3. Compute inner hash: inner = H( (K ⊕ ipad) || m )
4. Compute outer hash: HMAC = H( (K ⊕ opad) || inner )

Why HMAC prevents tampering:

1. Secret Key Requirement:
   - Attacker does NOT know key K
   - Cannot compute correct HMAC for any message without K
   - Modifying m changes HMAC in unpredictable way for non-key-holder
   - Even one-bit change produces completely different HMAC

2. Two-layer Design:
   - Outer hash uses key-masked input
   - Inner hash uses different key-masking
   - Prevents simple key recovery or extension attacks
   - Even if inner hash leaked, outer hash still requires key

3. Resistance to Length Extension Attacks:
   - Plain SHA-256 has length extension vulnerability:
     * Given H(m), one can compute H(m||padding||x) without knowing m
     * This breaks many simple MAC constructions like H(key||message)
   - HMAC uses double hashing to prevent length extension:
     * Inner hash hides message structure
     * Outer hash adds final protection

Attack Scenarios Comparison:

Scenario 1: Plain hash for file integrity
- Site publishes file + hash value
- Attacker intercepts download, modifies file, recomputes new hash
- User checks hash, sees matching value -> tampered file accepted

Scenario 2: Plain hash with secret suffix: H(m||K)
- Attacker intercepts H(m||K)
- Uses length extension attack to compute H(m||padding||x||K)
- Can append data without knowing K -> breaks authentication

Scenario 3: Plain hash with secret prefix: H(K||m)
- Vulnerable to collision attacks if hash is broken
- Attacker can find m1 ≠ m2 with same H(K prefix) after analysis

Scenario 4: HMAC (secure)
- Attacker captures HMAC(K, m) for some m
- Cannot compute HMAC for any different m' without K
- Length extension impossible due to double structure
- Even if hash is weak (MD5), HMAC remains secure if key is long enough

HMAC Security Properties:

1. Message Integrity:
   - Any change to m produces different HMAC with probability ~1
   - Receiver detects mismatch: tampered message

2. Message Authentication:
   - Only sender with key K can produce valid HMAC
   - Receiver knows message originated from legitimate source
   - Also called "origin authentication"

3. Keyed Property:
   - Different keys produce completely different HMACs for same message
   - Keys must be stored securely

HMAC vs Digital Signatures:
+-------------------+------------------------+------------------------+
| Property          | HMAC                   | Digital Signature      |
+-------------------+------------------------+------------------------+
| Uses symmetric key| Yes                    | No (asymmetric)        |
| Non-repudiation   | No (sender can deny)   | Yes (unique to signer) |
| Key distribution  | Secure channel needed  | Public keys available  |
| Speed             | Very fast              | Slower                 |
| Use case          | Secure channels (TLS)  | Signatures, PKI        |
+-------------------+------------------------+------------------------+

Real-world usage:
- TLS/SSL record layer authentication
- IPSec AH/ESP authentication
- AWS API request signing
- OAuth 2.0 (HMAC-SHA256)
- JWT (HMAC-SHA256 signatures)

PAPER 13
Theory: Explain how digital signatures ensure integrity and non-repudiation.

Answer (Expanded):

Digital signatures are the cryptographic equivalent of handwritten signatures,
but they provide much stronger security guarantees. They simultaneously provide
integrity, authentication, and non-repudiation using asymmetric cryptography.

Core Mechanism (RSA-based signature):

Signing process (private key):
1. Compute hash of message: h = H(m) (e.g., SHA-256)
2. Encrypt hash with signer's private key: s = h^d mod n
3. Output (m, s) as signed message

Verification process (public key):
1. Compute hash of received message: h' = H(m')
2. Decrypt signature with signer's public key: h'' = s^e mod n
3. Accept if h' == h'', else reject

How Integrity is Achieved:

1. Any modification to m changes computed hash h'
2. Signature s is tied to original hash h, not the modified message
3. After modification: h' != h → verification fails
4. This works even for single-bit changes due to avalanche effect
5. Attacker cannot "fix" signature without signer's private key

Example:
- Original: m = "Pay Alice $100", H(m) = 0xA1B2...
- Signature = encrypt(0xA1B2..., private_key)
- Attacker changes to "Pay Eve $1000", H(m') = 0xFFFF...
- Decrypted signature gives 0xA1B2... ≠ 0xFFFF... → reject

How Non-repudiation is Achieved:

1. Private key is known only to the signer (properly protected)
2. Public key is widely available and associated with signer's identity (via certificate)
3. Anyone can verify signature using signer's public key
4. If verification passes, only the private key holder could have created it
5. Signer cannot later deny signing without claiming their private key was compromised

Important nuance:
- Non-repudiation in cryptography is mathematical, but legal non-repudiation
  requires additional infrastructure (PKI, timestamping, audit trails)
- If signer's private key was stolen, they can repudiate by proving compromise
- Hence hardware security modules (HSMs) and secure key storage are critical

Security Properties Breakdown:

+-----------------------+------------------------------------------------+
| Property              | How Digital Signature Provides It            |
+-----------------------+------------------------------------------------+
| Integrity             | Hash mismatch reveals any message change     |
| Authentication        | Signature proves origin (signer's private key)|
| Non-repudiation       | Third parties can verify without signer help |
| Unforgeability        | Cannot create valid signature without private key|
| Verifiability         | Anyone with public key can verify             |
+-----------------------+------------------------------------------------+

Comparison with Other Mechanisms:

Plain Hash (no signature):
- Integrity: Yes (if attacker doesn't recompute hash)
- Authentication: No
- Non-repudiation: No
- Vulnerability: Attacker can recompute hash after modification

HMAC (symmetric):
- Integrity: Yes
- Authentication: Yes (shared secret)
- Non-repudiation: No (both parties know same secret)
- Vulnerability: Receiver could forge sender's signature

Digital Signature (asymmetric):
- Integrity: Yes
- Authentication: Yes (unique private key)
- Non-repudiation: Yes
- Vulnerability: Private key theft (solve with HSMs)

ECDSA vs RSA Signatures:
+----------------+-----------------------+-----------------------+
| Aspect         | RSA                   | ECDSA                 |
+----------------+-----------------------+-----------------------+
| Key size       | 2048-4096 bits        | 256-521 bits          |
| Signature size | Same as key size      | ~64 bytes for 256-bit |
| Speed          | Moderate              | Fast (more efficient) |
| Security level | High                  | Equivalent strength   |
| Popularity     | Most common (PKI)     | Growing (Blockchain)  |
+----------------+-----------------------+-----------------------+

Real-world Applications requiring non-repudiation:
- Software code signing (Microsoft Authenticode, Apple notarization)
- Legal contracts (DocuSign, Adobe Sign with digital certificates)
- Financial transactions (SWIFT, credit card authorizations)
- TLS server certificates (bind domain name to public key)
- Blockchain transactions (Bitcoin, Ethereum using ECDSA)
- Git tag signing (developer authenticity)

- Does NOT provide confidentiality (message is sent in plaintext)
- For confidentiality, use sign-then-encrypt or encrypt-then-sign
- Public key must be trusted (PKI or web of trust required)

PAPER 14
Theory: Explain why frequency distribution makes monoalphabetic cipher insecure.

Answer (Expanded):

Monoalphabetic substitution ciphers replace each plaintext letter with a fixed
ciphertext letter according to a permutation of the alphabet. Despite having
26! ≈ 2^88 possible keys (more than enough for exhaustive search in theory),
the cipher is easily broken using frequency analysis because it preserves all
statistical properties of the underlying language.

Why Frequency Distribution Remains Preserved:

In English and most natural languages:
- Letters occur with characteristic frequencies
- These distributions are stable across most texts
- The permutation shuffles which ciphertext letter corresponds to which frequency,
  but the distribution pattern itself remains recognizable

English letter frequencies (approximate):


Frequency Attack Steps:

Step 1: Count ciphertext letters
- Suppose ciphertext contains: X appears 127 times, Q appears 90 times, etc.
- Sort by frequency

Step 2: Map most frequent ciphertext letter to 'E'
- If X appears most often (12% of text), guess X = E

Step 3: Map next frequent letters
- Second most frequent → likely 'T'
- Third → 'A' or 'O', etc.

Step 4: Use digraph and trigraph patterns
- Common digraphs: TH, HE, IN, ER, AN, RE, ED, ON, ES, ST
- Common trigraphs: THE, AND, ING, ION, TIO, FOR, NDE, HAS, NCE
- Example: If ciphertext contains frequent "XYZ" and X=E, Y may be 'H', Z='E'? Wait, Z would then match E again — impossible.
- Use patterns to resolve ambiguities

Step 5: Guess short common words
- Three-letter words: "THE", "AND", "FOR", "ARE"
- Two-letter words: "OF", "TO", "IN", "IT", "IS", "ON"
- Fill in mappings from word guesses

Step 6: Iterate refinement
- Use partial decryption to guess more words
- Adjust conflicting mappings
- Complete full decryption

Why this works even with long keys:
1. Natural languages are highly redundant (entropy ~1-2 bits per letter vs 4.7 bits random)
2. Ciphertext preserves all statistical patterns
3. Even if mapping is unknown, distribution reveals it
4. Length of text needed: ~50-100 characters sufficient for English

Modern computer attack speed:
- English frequency analysis: <1 millisecond for 1000-character text
- Does not require brute force over 26! keys
- Automated pattern matching and dictionary validation

Expanded Attack: Index of Coincidence (IC):
- Measures probability two random letters from text are equal
- For English: IC ≈ 0.066 (expected if random letters: 1/26 = 0.0385)
- For monoalphabetic cipher: IC remains 0.066 (preserved)
- This confirms it's a substitution cipher before any decoding

Example Attack Demonstration:

Ciphertext (short): "GUR DHVPX OEBJA SBK WHZCF BIRE GUR YNML QBT"
Frequency: G appears 4 times (most), R appears 4 times (most), etc.
Step: Guess G=E (common), UR = HE? U=H, R=E?
Then GUR = THE, R=E, U=H, G=T confirms.
"THE Q___X ..." continue attacking.

Why Monoalphabetic Ciphers are Completely Insecure:

1. Preserves letter frequencies (exposed)
2. Preserves digraph/trigraph patterns (exposed)
3. Preserves word lengths (exposed)
4. No confusion/diffusion per modern cryptography
5. Known plaintext attack trivially recovers full key
6. Chosen plaintext attack trivial (just ask to encrypt "ABCDEFGHIJKLMNOPQRSTUVWXYZ")

Cryptographic Lessons Learned:
- Substitution alone is insufficient for security
- Need both substitution AND permutation (like in AES)
- Modern ciphers operate on bits, not letters
- Must break statistical patterns through diffusion

Comparison to Modern Ciphers:
+--------------------+----------------+------------------+
| Aspect             | Monoalphabetic | AES-256          |
+--------------------+----------------+------------------+
| Key space          | 2^88           | 2^256            |
| Frequency hides?   | No             | Yes              |
| Known plaintext    | Trivial        | Infeasible       |
| Time to break      | Milliseconds   | Beyond universe  |
| Use today          | Puzzles only   | Global standard  |
+--------------------+----------------+------------------+

Real-world historical note:
- Monoalphabetic ciphers broken in 9th century (Al-Kindi)
- Used through 18th/19th centuries but increasingly insecure
- Became obsolete after World War I with advanced cryptanalysis
- Only present in puzzles and children's cryptography today

PAPER 15
Theory: Explain how key length affects security in Vigenere cipher.

Answer (Expanded):

The Vigenere cipher is a polyalphabetic substitution cipher that uses a keyword
to determine shifting of letters. Unlike monoalphabetic ciphers, different
positions of plaintext are shifted by different amounts based on the keyword
letter at that position.

Vigenere Encryption Formula:
- Ci = (Pi + Ki) mod 26
- Where K is keyword repeated: Ki = keyword[i mod key_length]

Key Length (L) Determines Periodicity:

For key length L, the cipher repeats every L characters:
- Positions 1, L+1, 2L+1, ... all use the same shift value
- Positions 2, L+2, 2L+2, ... use next shift value, etc.
- This creates L independent interleaved Caesar ciphers
- Each Caesar cipher is a simple shift (monoalphabetic within its position set)

How Longer Key Length Improves Security:

1. Brute force complexity:
   - Number of possible keys for length L: 26^L
   - L=4: 456,976 keys (trivial to brute force)
   - L=6: 308 million keys (moderate, possible on PC)
   - L=8: 208 billion keys (harder but possible with distributed computing)
   - L=12: 9.5 × 10^16 keys (impractical for brute force on classical computers)
   - L=20: 2 × 10^28 keys (impossible to brute force)

2. Period length effect on Frequency Analysis:
   - For short key (L=3): each position gets many samples (text_length/3)
   - Enough data to run separate frequency analysis on each position
   - Attack: Determine L first (Kasiski), then break L independent Caesar ciphers
   - For longer L (e.g., L=20): each position gets text_length/20 samples
   - Short texts may have insufficient samples per position
   - Attack requires longer ciphertext to succeed

3. Kasiski Examination becomes harder:
   - Repeated patterns in ciphertext occur when same plaintext aligns with same key positions
   - Distance between repeats is multiple of key length
   - Longer key length = more spaced repeats = harder to identify
   - But still possible with sufficient ciphertext

Index of Coincidence (IC) and Key Length:
- For English plaintext: IC ≈ 0.066
- For Vigenere with correct key: IC ≈ 0.066 (decrypted)
- For incorrect key guess: IC ≈ 0.038 (random)
- For partially correct key: IC between the two
- IC for ciphertext with L=4: IC ≈ 0.045 (some structure remains)
- IC for ciphertext with L=20: IC approaches 0.038 (more random-looking)

Quantitative Security Analysis:

Given ciphertext of length N and key length L:
- Samples per position: N/L
- Minimum N needed for reliable frequency attack: roughly 500 / L
- For L=4: need ~125 characters to break
- For L=10: need ~500 characters to break
- For L=20: need ~1000 characters to break
- For L=50 with N=500: only 10 samples per position — frequency analysis unreliable
- For L = N (running key): each position used only once — no repeat — one-time pad security

Limitations:
1. If key length < message length sufficiently, attack is still possible
2. Kasiski examination can recover L from repeated patterns
3. Friedman test using IC can also estimate L
4. Known plaintext attack trivially recovers key regardless of length
5. Once L is known, cipher reduces to L independent Caesar ciphers

Maximum Security: Running Key (L = N)
- Key length equals message length
- Each position gets unique shift (no repetition)
- Becomes equivalent to One-Time Pad if:
  * Key is truly random (not derived from a word)
  * Key never reused
  * Key distribution is secure
- Historical "Vigenere with running key" was used but often with predictable keys
  (book passages, common phrases) which were breakable

Comparison Table:
+--------------+--------------+--------------------+-------------------------+
| Key Length L | Brute Force  | Freq analysis      | Practical breakability  |
+--------------+--------------+--------------------+-------------------------+
| 1 (Caesar)   | 26 trials    | Trivial (single)   | Instant                 |
| 3-5          | 10^4-10^7    | Very easy (300+ ch)| Seconds                 |
| 6-8          | 3×10^8-2×10^11| Easy (500+ ch)     | Minutes/hours           |
| 10-12        | 10^14-10^17  | Moderate (1000+ ch)| Days (automated)        |
| 15-20        | 10^21-10^28  | Hard (5000+ ch)    | Impractical (manual)    |
| Running Key  | 26^N (huge)  | Impossible (N=len) | Secure (if random key)  |
+--------------+--------------+--------------------+-------------------------+

Example Attack on Short Key:

Plaintext:  "THIS IS A SECRET MESSAGE"
Key:        "KEYKEYKEYKEYKEYKEYKEY"
Key length L=3

Ciphertext positions:
Pos 1,4,7,10,13,16,19: shift by K (10)
Pos 2,5,8,11,14,17,20: shift by E (4)
Pos 3,6,9,12,15,18,21: shift by Y (24)

Attack steps:
1. Detect L=3 via Kasiski (look for repeated ciphertext trigrams)
2. Group positions 1,4,7,... and do frequency analysis → solve shift = 10 → K
3. Repeat for other positions

Lesson for Modern Cryptography:
- Repetition is dangerous in any cipher (ECB repeat blocks, Vigenere period)
- Need to avoid any deterministic repetition with same key
- Modern modes (CBC, CTR, GCM) use IV/nonce to break repetition
- Block cipher mode design respects this lesson from Vigenere

Theory: Explain how data is encapsulated across layers in TCP/IP during

Answer:

Encapsulation wraps data from higher layers with headers from lower layers as it
moves down the TCP/IP stack. Each layer treats the entire upper layer data as its

- Adds Ethernet header (MAC addresses) + FCS trailer

Layer 1 - Physical:
- Converts frame to bits on wire

[App Data] → TLS encrypt → [TCP Header | Data] → [IP Header | TCP | Data] →

TLS decrypts at transport/application boundary, revealing the HTTP request.

Key insight: Link layer doesn't know it's carrying TCP; network layer doesn't
know data is encrypted. This modularity allows different technologies to work

PAPER 20
Theory: Explain how ARP poisoning leads to man-in-the-middle attacks.

Answer:

ARP (Address Resolution Protocol) maps IP addresses to MAC addresses on a local
network. ARP poisoning (spoofing) is when an attacker sends forged ARP messages
to associate their own MAC address with another host's IP (e.g., the gateway).

How ARP poisoning creates MITM:

Step 1: Attacker M joins the local network
Step 2: M sends spoofed ARP to Victim A: "Gateway IP is at MAC_M"
Step 3: M sends spoofed ARP to Gateway G: "Victim A's IP is at MAC_M"
Step 4: Both A and G now send traffic intended for each other to M
Step 5: M forwards traffic after sniffing/modifying it

Result: All traffic between A and G flows through M:

Normal:   A <----------> G
After:    A <----> M <----> G   (M is in the middle)

M can now:
- Sniff all traffic (passwords, messages, files)
- Modify traffic (inject malware, redirect downloads)
- Drop or delay traffic (DoS)
- Strip TLS warnings (if user ignores certificate errors)

Detection methods:
- Static ARP entries (manual, not dynamic)
- ARP spoofing detection tools (arpwatch)
- DAI (Dynamic ARP Inspection) on managed switches

Prevention:
- Use encryption (TLS, SSH, VPN) — sniffed data remains unreadable
- Enable DAI on enterprise switches
- Use static ARP for critical hosts (gateway, DNS)

Critical fact: ARP poisoning works only on the local network segment, not over
the internet (routers don't forward ARP broadcasts).
